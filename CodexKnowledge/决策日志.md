## 2026-06-26：确立通用模块化架构（v4.0 方向）

**决策**：SCX v4 进行通用模块化重构，核心是"抽象接口 + Encoder 插件 + YAML 配置"三层分离。

**来源**：GPT 讨论 (`与gpt的对话202606261332.md`) 中用户指出"框架还是需要更通用一些，避免一直加模块迭代麻烦，应该模块化，不同的行业可以调用相同的代码"。

**核心设计**：

1. **SCXStateEncoder / SCXExpert / SCXDataPoint 抽象基类** — 所有核心模块只依赖抽象接口
2. **Encoders 插件式** — 每个领域继承抽象基类，实现自己的 encoder
3. **Config 声明式** — YAML 描述完整任务，`scx run --config domains/mlip.yaml`

**四个阶段**：
- Phase A (2-4周, P0): 核心重构，336 tests 通过
- Phase B (2-3周, P1): 声明式配置 + CLI
- Phase C (4-8周, P1): 新领域扩展（Robot/FEM/LLM）
- Phase D (长期, P2): 插件系统 + Benchmark + 社区

**Why**：
- 每加一个领域写一个 adapter，代码重复严重
- 接口不统一（MLIP encoder 和 Vision encoder 完全不同）
- 配置散落各处，无法通过 YAML 完整描述一个 SCX 任务
- 商业上无法快速生成"试点→审计报告"的标准流程
- 重构后新增领域的适配成本从"2-3 周"降到"2-3 天"

**关键文件**：
- `SCX_规划_v4_通用模块化.md` — 完整规划
- `SCX_TODO_v4_通用模块化.md` — 详细待办（替代 `SCX_TODO_2026-06-26.md`）

## 2026-06-27：定理证明里程碑

### 决策 1：方案 D — Theorem 1+2 两阶段发布

**决策**：Theorem 1 (Noise Detection) 和 Theorem 2 (Weak Feature Failure) 采用两阶段发布策略：arXiv 先发草稿版 → 经同行审查后发修订版。

**Why**：
- 两个定理的数学根基已经扎实（Chernoff bound、Fano inequality）
- 但需要更多实验验证才能确保 publishing-ready
- arXiv 占坑优先于完美

### 决策 2：A6 假设添加到 Theorem 1

**决策**：为保证 Theorem 1 的方向正确，新增假设 A6：专家在噪声样本上的一致性显著高于在清洁样本上的一致性。

**Why**：
- Lemma 3 的原结构缺少方向性保证
- A6 使 Chernoff bound 的 KL 散度方向判断成立
- 该假设在实践中有直观合理性（多头对同一错标达成一致的概率高于随机）

### 决策 3：V(s) 公式 deprecated

**决策**：旧数据价值公式 V(s) = rho · r_bar · L · (1-D) · max SCX 标记为 deprecated，不再推荐使用。

**Why**：
- 该公式本质是启发式（heuristic），缺乏严格的定理保证
- Theorem 1-3 提供了更well-defined的替代方案
- 保留代码兼容但文档中标记为 deprecated
- 未来版本可能完全移除
