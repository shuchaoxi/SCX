# 第一轮审查（CYCLE 1）：问题发现

---

## 一、事实性与表述错误

| # | 位置 | 原文 | 问题 | 严重程度 |
|---|------|------|------|---------|
| 1 | 原L29 | "审核新维护者" | 应为"审核新维护者候选人"。原文暗示内核直接审核"维护者"，但候选人尚未成为维护者，术语不一致 | 低 |
| 2 | 原L36 | "全票或否决" | 表述模糊。"全票"和"否决"是同一逻辑的两面（全票通过 = 无一否决），用"或"连接造成歧义 | 中 |
| 3 | 原L65 | "GitHub (已有)" | 括号内"已有"风格不一致——其他单元格均为功能性描述而非状态描述 | 低 |
| 4 | 原L67 | "Discord 或 Matrix" | 二选一未做决定。作为方案文档应给出明确选择及理由，而非保留选项 | 中 |
| 5 | 原L85 | "你和梗同学 + 1-2 个被审计通过的人" | 内部文档中出现具体人名和"你"（第二人称），不符合正式方案文档规范。内容泄露个人信息，且不可泛化 | 高 |
| 6 | 原L92 | "社区不是你的……你只是第一个维护者……你接受这一点" | 整段使用"你"称呼，面向特定读者而非作为制度文档。语气从方案说明降格为个人劝诫 | 中 |

---

## 二、结构性缺失

| # | 缺失内容 | 为什么重要 | 严重程度 |
|---|---------|-----------|---------|
| 1 | **冲突解决机制** | 内核采用全票通过制，必然存在僵局风险。无僵局处理流程 = 决策机制不完整。一个否决就能永久阻塞所有决策 | **极高** |
| 2 | **紧急程序** | 如果维护者被检测到 g≠0、密钥泄露、或被胁迫——没有任何驱逐/替换程序。这在治理上等同于承认"恶意维护者无法被移除" | **极高** |
| 3 | **g=0 声明规范** | 文档反复提及"公开 g=0 声明"但从未定义声明应包含什么内容、什么格式、如何验证。没有标准化的声明模板，g=0 就是一个空口号 | **极高** |
| 4 | **资金来源的量化** | "Yajie API 校准费"作为资金来源——但 Yajie API 目前不存在。没有估算年度运维成本、初始资金需求、或资金来源的可行性 | 高 |
| 5 | **任期长度与轮换频率** | "轮换池"和"定期重新审计"被提及但从未定量——多长时间一轮？任期多久？没有时间参数，轮换就无法执行 | 中 |
| 6 | **贡献者→内核的晋升量化标准** | "至少一次有效贡献被内核接受"是贡献者门槛——但晋升内核需要多少贡献？审计频率？无量化 | 中 |

---

## 三、逻辑与概念问题

| # | 问题 | 分析 | 严重程度 |
|---|------|------|---------|
| 1 | **"Yajie API 校准锚点"未定义** | L29 提到内核维护者"轮换维护 Yajie API 校准锚点"，但全文从未解释什么是校准锚点、如何校准、为什么需要维护 | 中 |
| 2 | **审计日志的"不可篡改"声明** | L69 称审计日志存储在"公开 Git 仓库 (不可篡改)"——Git 历史可通过 force push 改写，并非不可篡改。应说明附加措施（如多节点镜像、区块链锚定或时间戳服务） | 低 |
| 3 | **"Signal 群组"作为内核私密讨论的唯一载体** | Signal 是端到端加密的，但群组管理依赖 Signal 的服务器。如果 Signal 被攻破或关闭，内核通信即中断。缺少备用方案 | 低 |

---

## 四、修复摘要

CYCLE 1 完成后执行以下修复：

1. **修正术语**: "审核新维护者" → "审核新维护者候选人"，增加"（CEC 基准值）"说明校准锚点
2. **澄清投票规则**: "全票或否决" → "一致性规则：全票通过方可加入；任一否决即不通过"
3. **统一风格**: "GitHub (已有)" → "GitHub（现有仓库）"；"Discord 或 Matrix" → "Matrix（开源、去中心化）"（选择 Matrix 因其开源去中心化与 SCX 理念一致）
4. **去人称化**: 所有"你"/"你和梗同学"替换为中性制度表述
5. **新增"冲突解决"章节**: 三层升级路径（内核讨论→贡献者建议→独立审计）
6. **新增"紧急程序"章节**: g≠0 检测→48h 审计→移除→替补的完整流程
7. **新增"g=0 声明规范"章节**: 五项必须内容（身份、弃权、审计同意、轮换同意、GPG 签名）
