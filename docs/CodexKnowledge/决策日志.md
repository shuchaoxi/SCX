## 2026-06-26：确立通用模块化架构（v4.0 方向）

**决策**：SCX v4 进行通用模块化重构，核心是"抽象接口 + Encoder 插件 + YAML 配置"三层分离。

**来源**：GPT 讨论 (`与gpt的对话202606261332.md`) 中用户指出"框架还是需要更通用一些，避免一直加模块迭代麻烦，应该模块化，不同的行业可以调用相同的代码"。

**核心设计**：

1. **SCXStateEncoder / SCXExpert / SCXDataPoint 抽象基类** — 所有核心模块只依赖抽象接口
2. **Encoders 插件式** — 每个领域继承抽象基类，实现自己的 encoder
3. **Config 声明式** — YAML 描述完整任务，`scx run --config domains/mlip.yaml`

**四个阶段**：
- Phase A (2-4周, P0): 核心重构，336 tests 通过
- Phase B (2-3周, P1): 声明式配置 + CLI
- Phase C (4-8周, P1): 新领域扩展（Robot/FEM/LLM）
- Phase D (长期, P2): 插件系统 + Benchmark + 社区

**Why**：
- 每加一个领域写一个 adapter，代码重复严重
- 接口不统一（MLIP encoder 和 Vision encoder 完全不同）
- 配置散落各处，无法通过 YAML 完整描述一个 SCX 任务
- 商业上无法快速生成"试点→审计报告"的标准流程
- 重构后新增领域的适配成本从"2-3 周"降到"2-3 天"

**关键文件**：
- `SCX_规划_v4_通用模块化.md` — 完整规划
- `SCX_TODO_v4_通用模块化.md` — 详细待办（替代 `SCX_TODO_2026-06-26.md`）

## 2026-06-28：论文 LaTeX 编译环境搭建与修复

### 决策 1：VS Code LaTeX 插件从 vscode-latex-runner 迁移到 LaTeX Workshop

**决策**：卸载 `finlay-ab.vscode-latex-runner`，安装 `james-yu.latex-workshop` v10.16.1。

**Why**：
- `vscode-latex-runner` 只能手动编译（Ctrl+Alt+B），无法保存即编译
- LaTeX Workshop 是 VS Code 标准 LaTeX 插件（10M+ 安装量），支持 auto-build on save、内置 PDF 预览、SyncTeX
- 同时卸载冲突的 `mathematic.vscode-pdf`（与 LaTeX Workshop 争抢 PDF 文件打开权）

### 决策 2：编译方案选择 pdflatex x 3（非 latexmk）

**决策**：默认编译方案使用三次 `pdflatex`（直接调用），不使用 `latexmk` 包装。

**Why**：
- 当前 MiKTeX 缺少 Perl 运行环境，`latexmk` 不可用
- `pdflatex` 是 APS/revtex 期刊的标准编译引擎
- 三次编译确保交叉引用和参考文献正确解析
- 禁用 autoClean（同样依赖 latexmk）

**后续**：如需安装 Perl 恢复 latexmk，MiKTeX 控制台可一键安装。

### 决策 3：修复论文代码中的 TeX 非法命令名

**决策**：将 `egp_merging/main.tex` 中 `\c0` 全文重命名为 `\cb`（25 处），避免与 TeX 内置 `\c`（cedilla）命令冲突。

**Why**：
- TeX 控制序列名只能包含字母（A-Z a-z）
- `0` 不是字母，`\c0` 被 TeX 解析为 `\c` + 字符 `0`
- 这个问题在 Overleaf（TeX Live）上不报错但本地 MiKTeX 报错——编译器差异

### 决策 4：宏双下标问题采用直接展开策略

**决策**：`\cZ_i` 等 6 处 double subscript 不修改宏定义，改为直接展开为 `\mathbf{c}_{Z_i}`。

**Why**：
- 修改宏定义为接受可选参数会引入新的复杂性（`\cZ` 本身与 `\c` 的命名冲突）
- 6 处改动量小，直接展开最安全
- 保留宏在无需物种下标的位置（如 `\cZ^{(m)}`）继续使用

### 决策 5：确立"作者模式 + 审稿模式"双轨验证方法论

**决策**：Hermes（6/27）完成证明后，由 Claude Code（6/28）启动 5-agent 独立对抗性审查，不信任 Hermes 的"全修好了"结论。

**Why**：
- Hermes 工作在"证明模式"（作者视角），目标是完成推导、修复已知 bug
- Claude Code 工作在"审稿模式"（reviewer 视角），目标是找到拒稿理由
- 两种模式的认知偏差不同：作者倾向于确认自己是对的，审稿人倾向于假设作者是错的
- 5 个 agent 互不通信、独立出报告，防止群体思维
- 结果验证了方法论：核心定理（Thm 1-3）PASS，但 SE-1 和 Prop 3 在 hostile review 下暴露了之前未发现的硬伤

**经验**：
- "全修好了"从作者视角是诚实的——Hermes 确实修了当时已知的所有问题
- 独立对抗性审查发现了 35 个新缺陷——这不是 Hermes 的失误，而是独立审查的价值
- 学术投稿前，两种模式缺一不可：先作者模式完成推导，再审稿模式逐条攻击
- 这个方法应该固化为 SCX 项目的标准流程：任何投稿前都跑一轮 multi-agent hostile review
