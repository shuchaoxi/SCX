# SCX 内河社区方案

**日期**: 2026-07-02
**分类**: 内部

---

## 为什么需要社区

定理、代码、论文——都有了。缺一群人。不是用户——是**维护者**。TCP/IP 没有社区就死了。Linux 没有社区就死了。SCX 需要一群人——不是崇拜者，是审计者。不是粉丝，是维护者。

## 社区定位

**非盈利。开源。无所有权。**

| 不是 | 是 |
|------|------|
| 公司 | 协议维护网络 |
| 粉丝群 | 审计者社区 |
| 付费会员制 | 开源贡献者网络 |
| 你的个人品牌 | 定理的共同守护者 |

## 社区结构

### 三层

| 层 | 谁 | 做什么 | 门槛 |
|:--:|------|------|------|
| **内核 (2-5人)** | 初始维护者 | 轮换维护 Yajie API 校准锚点（CEC 基准值）。审核新维护者候选人。 | 通过 M>1 审计 + 公开 g=0 声明 |
| **贡献者 (~20人)** | 活跃开发者、审稿人 | 提交代码、审查论文、发现漏洞、复现审计结果 | 至少一次有效贡献被内核接受 |
| **观察者 (不限)** | 任何人 | 使用 Spring、跑 Yajie、提 issue、讨论 | 零 |

### 加入内核的路径

```
贡献者 → 公开声明 g=0 → 接受 M>1 审计 
→ 内核投票（一致性规则：全票通过方可加入；任一否决即不通过）
→ 进入轮换池 → 定期重新审计
```

## 讨论内容

### 内核讨论（私有）
- 维护者轮换日程
- 新维护者审核
- CEC 校准异常调查
- 审计标准修订提案

### 贡献者讨论（半公开）
- 代码审查
- hostile review 论文
- 实验复现
- 新方向提案

### 公开讨论
- 使用问题
- Cercis 分数比较
- 审计日志讨论
- 任何声称的独立验证

## 冲突解决

内核决策原则：全票通过制。若出现僵局（deadlock），触发以下升级路径：
1. 内核内部第二轮讨论（延长讨论期 72 小时）
2. 若仍僵局：议题降级至贡献者层公开讨论，由贡献者投票提供建议（非约束性）
3. 若涉及 g=0 判定：要求被质疑方接受独立第三方 M>1 审计

## 紧急程序

若内核维护者被检测到 g≠0 或出现恶意行为：
1. 任何贡献者可触发"紧急审计"——公开提交 g≠0 的证据
2. 其余内核成员 48 小时内启动 M>1 独立审计
3. 确认 g≠0 后：立即移出内核 + 公开审计日志
4. 从贡献者轮换池中提名替代者

## g=0 声明规范

候选人提交的 g=0 公开声明须包含：
- 身份声明（实名或可验证的长期化名）
- 明确放弃对 SCX 协议的任何所有权主张
- 同意接受定期 M>1 审计，审计日志完全公开
- 同意轮换机制——任期结束后自动退出内核（可重新申请）
- 声明文件须签名（GPG），发布于公开 Git 仓库

## 第一步

1. 创建 Signal 群组——初始内核（2-5 名通过审计的初始维护者）
2. 公开"维护者候选人声明"模板——任何人可以提交 g=0 声明
3. 第一次公开 hostile review 活动——悬赏找 SCX 论文的漏洞
4. 建立轮换日程——初始维护者的第一轮任期和审计时间表

## 核心原则

> **社区不是任何个人的——社区是定理的。** 初始维护者只是第一批守护者。定理不依赖任何个人。任何时候社区可以审计维护者、替换维护者。接受这一点是所有维护者的基本条件——否则就不满足 g=0。
